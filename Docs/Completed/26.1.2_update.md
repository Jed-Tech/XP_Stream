# Minecraft 26.1.2 Update Plan

Plan for moving the workspace from the current `26.1.1` pins to the `26.1.2` line, updating **NeoForge** to `26.1.2.7-beta`, and preparing **Fabric** for Minecraft `26.1.2` while keeping broad compatibility policy intact.

**Status:** Executed (2026-04-11)  
**Date:** 2026-04-11

---

## Goals

- Update repo-wide development pins to the `26.1.2` line.
- Move **NeoForge** from `26.1.1.1-beta` to `26.1.2.7-beta`.
- Prepare **Fabric** to build and run against `26.1.2`.
- Keep both mods buildable:
  - `xp_stream` on Fabric + NeoForge
  - `saturation_regen` on Fabric
- Preserve the standing broad compatibility policy in [Docs/broad_minecraft_family_policy.md](C:\Users\jedde\GithubRepos\Minecraft\XP_Stream\Docs\broad_minecraft_family_policy.md).

## Planned changes

### 1. Repo-wide pins

Update [gradle.properties](C:\Users\jedde\GithubRepos\Minecraft\XP_Stream\gradle.properties):

- Set `minecraft_version=26.1.2` if it resolves in Fabric Loom
- Set `neoforge_version=26.1.2.7-beta`
- Re-check whether these Fabric-side pins can stay or need updates:
  - `fabric_loader_version`
  - `fabric_loom_version`
  - `fabric_api_version`
- Re-check whether `moddevgradle_version` still matches the new NeoForge target cleanly

### 2. Broad metadata policy review

Per [Docs/broad_minecraft_family_policy.md](C:\Users\jedde\GithubRepos\Minecraft\XP_Stream\Docs\broad_minecraft_family_policy.md), broad player-facing metadata should remain broad unless there is a real incompatibility.

Expected outcome:

- NeoForge `minecraft` dependency stays at `versionRange = "[26.1-,)"`
- NeoForge `neoforge` dependency continues using `versionRange = "[${neoforge_version},)"`
- Fabric `depends.minecraft` floor likely stays at `>=26.1-alpha.2`

Files to verify:

- `mods/xp_stream/neoforge/src/main/resources/META-INF/neoforge.mods.toml`
- `mods/xp_stream/fabric/src/main/resources/fabric.mod.json`
- `mods/saturation_regen/fabric/src/main/resources/fabric.mod.json`

`saturation_regen` NeoForge metadata is out of scope here until that loader implementation is actually developed.

### 3. Fabric preparation

Fabric needs to be prepared for `26.1.2`, not just left assumed-compatible.

Explicit checks:

- confirm `minecraft_version=26.1.2` resolves
- confirm current Loom line still works
- confirm loader pin is still acceptable
- confirm whether Fabric API needs a `26.1.2`-specific bump

If any of those fail, update the Fabric pins before smoke testing.

### 4. NeoForge upgrade

NeoForge work for this cycle is straightforward unless the toolchain disagrees:

- update `neoforge_version` to `26.1.2.7-beta`
- keep broad metadata policy intact
- verify ModDevGradle compatibility during build

### 5. Documentation / release notes

If the update is completed successfully:

- add a new top entry to `CHANGELOG.md`
- note tested compatibility for Fabric + NeoForge on `26.1.2`
- record any pin changes required to make Fabric or NeoForge pass
