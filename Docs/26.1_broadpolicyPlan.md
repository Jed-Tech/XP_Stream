## Goal: Broad 26.1 family compatibility (pre-3 through 26.1.x)

We want the *entire mod* (common logic + both loader jars) to declare compatibility with the whole Minecraft `26.1` family, including pre-releases / RCs, and not be brittle to NeoForge’s “snapshot”-flavored identifiers.

Key policy we’re applying:
- NeoForge `minecraft` dependency range should be broad: **`[26.1-,)`**
- Fabric should remain broad via its existing early pre-release floor (it already uses an early `>=26.1-alpha.2`-style constraint).
- `common` should remain loader-agnostic (no metadata/version gating needed there).

## Current state (what we’re changing)

- `mods/xp_stream/neoforge/src/main/resources/META-INF/neoforge.mods.toml`

Fabric (already broad enough in spirit):
- `mods/xp_stream/fabric/src/main/resources/fabric.mod.json`
  - Minecraft dependency is already broad: `">=26.1-alpha.2"`

Common:
- `mods/xp_stream/common` has no loader metadata / version gating that needs to be “range” aware.

## Planned metadata changes

### 1) NeoForge: widen Minecraft versionRange to `[26.1-,)`
File:
- `mods/xp_stream/neoforge/src/main/resources/META-INF/neoforge.mods.toml`
- `mods/saturation_regen/neoforge/src/main/resources/META-INF/neoforge.mods.toml`

Change:
- Update the `[[dependencies.${mod_id}]]` entry for `modId = "minecraft"`:
  - From: `[26.1-snapshot-1,)`
  - To:   `[26.1-,)`

Notes:
- We’re intentionally mirroring the “broad policy” used by projects like OkZoomer.
- We will keep the NeoForge dependency itself as “lower bound then open upper”:
  - `versionRange = "[${neoforge_version},)"`

### 2) Fabric: keep existing broad floor (don’t overfit labels)
File:
- `mods/xp_stream/fabric/src/main/resources/fabric.mod.json`

Decision:
- Keep `minecraft: ">=26.1-alpha.2"` unless runtime testing shows it rejects RC/pre builds.

Reason:
- Fabric already provides a very early compatibility floor, which matches the “broad policy” philosophy.
- This reduces the risk of accidentally using a version-range syntax/semantics Fabric doesn’t accept.

### 3) Common: no change
Files:
- `mods/xp_stream/common/...`

Decision:
- No metadata changes needed. Common stays as shared game logic.

## Version selection for development builds (what we will build/test)

We will build NeoForge against the latest NeoForge `26.1` prerelease available today (e.g. the Maven `<latest>` / `<release>` value), specifically including `...+pre-3` when it’s available.

Fabric:
- Build using whatever is currently in `gradle.properties` for `minecraft_version` (repo-wide dev pin set to the newest resolvable 26.1 pre-release, aligned to pre-3), but validate runtime against the same target builds we test on NeoForge.

---
## Verification matrix (runtime checks to ensure metadata actually matches)

We will run the *real game* with the broad policy to confirm the dependency checks accept the intended targets.

1) NeoForge: run client on NeoForge `26.1 ...+pre-3`
- Expectation: mod loads without “incompatible Minecraft version” / dependency rejection.
- Confirm functionality: XP orb burst pickup works.

2) NeoForge: run client on NeoForge `26.1 ...+pre-3` (or nearest newer prerelease/RC available at the time)
- Expectation: still accepted (broad range behaves as intended).

3) NeoForge: run client on NeoForge matching RC1 (once RC1 is available in Maven)
- Expectation: accepted + no mixin failures.

4) Fabric: run client on a matching 26.1 prerelease/RC build
- Expectation: accepted + no behavior regressions.

## Risk assessment & fallback plan

Risk:
- NeoForge version ranges mention that snapshots / pre-releases / RC may not sort “naturally” in Maven ranges.

Fallback:
- If NeoForge rejects a target despite `[26.1-,)`:
  - we will adjust the `minecraft versionRange` to the exact identifier used by NeoForge for `pre-3` (derived from the NeoForge MDK / run config metadata)
  - then keep the range open-ended above that identifier.

---
## Acceptance criteria

This broad policy plan is “done” when:
- NeoForge mod declares compatibility and successfully loads on `pre-3`.
- NeoForge mod successfully loads on RC1 when RC1 becomes available.
- Fabric continues to load and behave correctly across the same tested 26.1 prerelease/RC set.
