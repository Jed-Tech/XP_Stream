## 26.1 GA Release Plan (xp_stream only; Fabric first, then NeoForge)

This plan is for shipping a **GA (general availability) 26.1** release of the mods in this monorepo.

Repo facts (today):
- Fabric runtime gating already allows 26.1+ via `fabric.mod.json` (`minecraft >= 26.1-alpha.2`).
- Builds are currently pinned in root `gradle.properties` (e.g. `minecraft_version=26.1`, `neoforge_version=...+snapshot-7`).
- NeoForge runtime gating is currently snapshot-flavored (`versionRange = "[26.1-snapshot-1,)"`) in each NeoForge `neoforge.mods.toml`.

---

## Goals

- **Fabric first**: have an `xp_stream` Fabric JAR that is built/tested on **Minecraft `26.1` GA** and is ready to publish.
- **NeoForge second**: once NeoForge publishes a **26.1 GA** build, align to that bundle, update metadata gating appropriately, build/test, then publish.
- Keep changes **version/config only** unless GA forces code fixes.

---

## Scope / non-goals

- This plan is **xp_stream only**.
- `saturation_regen` is explicitly out of scope for this GA plan.

---

## Pre-flight checklist (do once)

- Ensure Java toolchain is still correct for 26.1 (this repo uses Java 25 via Gradle toolchain/root property).
- Clean working tree (ignore `.gradle/` and other local files in commits).
- Decide release version bump for `xp_stream` (per-mod `mod_version` in `mods/xp_stream/gradle.properties`).
- Branching strategy: **work from `main`** (no release branch). Tags optional.

---

## Compatibility & versioning policy (decide now, execute later)

- **Acceptance criteria (fast path):**
  - Game **launches cleanly** (no mixin apply failures / no startup errors attributable to `xp_stream`)
  - **200 XP orb test passes** (no XP loss; Mending preserved; config loads)
  - That’s it for the first GA release; more testing can happen after shipping.
- **Supported line:** **26.1.x**
  - We will claim support for **26.1.x** on hosting sites.
  - If a future hotfix breaks compatibility, we will adjust hosting-site metadata (and/or ship a hotfix) accordingly.

## Phase A — Fabric (priority)

### A1) Update target versions for GA

Files:
- `gradle.properties` (root)

Actions:
- Set `minecraft_version=26.1`
- Verify `fabric_loader_version` is still compatible (keep as-is unless GA requires a bump).
- Verify `fabric_loom_version` is still compatible (keep as-is unless Gradle fails to resolve/run).
- (Optional) If you ever re-enable Fabric API, verify `fabric_api_version` has a `+26.1` build that matches GA.

Notes:
- Fabric mod metadata is already permissive enough:
  - `mods/xp_stream/fabric/src/main/resources/fabric.mod.json`
  - both currently: `"minecraft": ">=26.1-alpha.2"`

### A2) Build Fabric artifacts

Commands (PowerShell examples):
- `.\gradlew :mods:xp_stream:fabric:build`
- (or build all) `.\gradlew build`

### A3) Smoke test on GA

Minimum:
- Launch Fabric dev run (if you use it) or run in a launcher with the built jar.
- Verify the mod loads and no mixin apply failures.

XP_Stream functional checks:
- Spawn a dense orb cluster (your “200 XP orbs” test).
- Confirm:
  - No XP loss
  - Mending still repairs (burst pickup uses vanilla path)
  - No noticeable “magnet” effect regression (still collision-based)
  - Config loads from `config/xp_stream.json`

### A4) Release metadata and publishing

Actions:
- Update changelog / release notes (at least “Tested on Minecraft 26.1 GA”).
- In Modrinth / CurseForge release UI (or your publish automation), **select Minecraft 26.1** as a supported game version.

Commands (if using your publish tasks):
- `.\gradlew :mods:xp_stream:publishFabric:modrinth :mods:xp_stream:publishFabric:publishToCurseForge`

Rollback if Fabric breaks:
- Revert root `minecraft_version` back to last known good snapshot/pre.
- If runtime is fine but compile breaks, bump **Loom** first, then **Loader**.

## Phase B — NeoForge (after Fabric is green)

NeoForge note:
- Even though `minecraft_version=26.1` is now set repo-wide for GA prep, **do not publish NeoForge yet** until:
  - NeoForge Maven has a **true 26.1 GA bundle** (B1/B2), and
  - We update `neoforge.mods.toml` using **MDK-derived** `versionRange`s (B3), and
  - NeoForge smoke test passes on GA.

### B1) Wait for NeoForge 26.1 GA build (do not guess)

We will **wait** until NeoForge Maven `maven-metadata.xml` shows a NeoForge build targeting/bundling **Minecraft `26.1` GA**.

Where to check (documented elsewhere in this repo):
- NeoForge Maven metadata (`maven-metadata.xml`) for latest/release:
  - `https://maven.neoforged.net/releases/net/neoforged/neoforge/maven-metadata.xml`
- NeoForge releases/news pages for which builds target which MC versions.

Outcome:
- Identify the NeoForge version string to put in `gradle.properties` once a 26.1 GA bundle exists.

### B2) Update pinned versions for GA

Files:
- `gradle.properties` (root)

Actions:
- Set `minecraft_version=26.1`
- Set `neoforge_version=<the GA-bundling NeoForge version>`
- Consider whether `moddevgradle_version` needs a bump for that NeoForge line (only if Gradle fails or NeoForge docs require it).

### B3) Fix NeoForge runtime Minecraft gating (important)

Current state:
- Each NeoForge `neoforge.mods.toml` uses:
  - `versionRange = "[26.1-snapshot-1,)"`

Why this matters:
- A GA Minecraft version string (`26.1`) is **not guaranteed** to satisfy a snapshot-flavored lower bound like `26.1-snapshot-1` under NeoForge/Forge version comparison rules.

Decision (locked): **MDK-derived ranges**

- We will generate/download the **NeoForge MDK for Minecraft 26.1 GA** and copy its dependency `versionRange` values verbatim into our `neoforge.mods.toml`.
- Rationale: lowest risk of “loader rejects mod” due to version parsing/comparison differences between snapshot/pre/GA strings.

Concrete steps:
- Get the NeoForge MDK for **Minecraft 26.1 GA** that matches the NeoForge line we’re targeting (B1/B2).
- In the MDK project, open the MDK’s `src/main/resources/META-INF/neoforge.mods.toml`.
- Copy the following ranges into our `xp_stream` NeoForge metadata:
  - `[[dependencies.<modid>]] modId="minecraft" versionRange = "..."`  ← copy exactly
  - `[[dependencies.<modid>]] modId="neoforge" versionRange = "..."`   ← copy exactly
- Paste them into:
  - `mods/xp_stream/neoforge/src/main/resources/META-INF/neoforge.mods.toml`

Notes:
- Keep the MDK TOML structure/keys intact; only replace our `versionRange` values with the MDK’s.
- This is compatible with our “26.1.x” support goal as long as the MDK range includes patch releases; verify that when copying.

Files to update:
- `mods/xp_stream/neoforge/src/main/resources/META-INF/neoforge.mods.toml`

### B4) Build NeoForge artifacts

Commands:
- `.\gradlew :mods:xp_stream:neoforge:build`
- (or build all) `.\gradlew build`

### B5) Smoke test on GA (NeoForge)

Minimum:
- Run NeoForge client and verify:
  - mod loads
  - mixins apply cleanly
  - XP burst behavior still correct

Rollback if NeoForge breaks:
- First confirm you have the correct NeoForge version that bundles MC 26.1 GA.
- If it’s correct and still fails, bump `moddevgradle_version` per NeoForge guidance.
- If runtime rejects the mod, revisit the `minecraft` `versionRange` in `neoforge.mods.toml` (B3).

---

## Final checklist before publishing

- `minecraft_version` is `26.1`
- Fabric JAR built + smoke tested on `26.1`
- NeoForge JAR built + smoke tested on `26.1` (with correct NeoForge bundle)
- Release notes mention:
  - “Tested on Minecraft 26.1 GA”
  - any loader minimums (Fabric Loader, NeoForge)
- Publishing game version selection includes `26.1` for each loader file/version
